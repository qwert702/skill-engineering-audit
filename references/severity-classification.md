# 严重等级分类指南

本文档定义审计发现项的严重等级判定标准。每个发现项必须按此标准分类。

---

## 等级定义

### 🔴 严重 (Critical)

**判定标准**：满足以下任一条件
- 直接导致安全漏洞，可被外部利用（SQL注入、命令注入、认证绕过、XSS）
- 导致敏感数据泄露（密码、密钥、个人身份信息、支付信息）
- 导致核心功能不可用或数据持久性损坏
- 可被用于远程代码执行、权限提升、越权访问
- 存在已知 CVE 且影响当前使用的版本

**响应时限**：24 小时内
**报告要求**：必须在执行摘要中高亮，需附复现步骤和修复方案

**典型示例**：
- 用户输入直接拼接到 SQL 查询中
- JWT Secret 硬编码在代码中
- 文件上传未做类型校验和路径安全检查
- 未授权接口可访问管理员功能
- 依赖中含有可远程利用的已知漏洞

### 🟠 高危 (High)

**判定标准**：满足以下任一条件
- 显著违反编码规范或架构原则，导致代码库维护成本急剧上升
- 导致非关键功能异常或性能显著退化
- 可能导致安全风险，但利用条件苛刻或需组合其他漏洞
- 大量代码重复或圈复杂度严重超标，影响可维护性
- 缺少必要的输入验证，但影响面有限

**响应时限**：1 周内
**报告要求**：需附影响评估和修复建议

**典型示例**：
- 函数圈复杂度 > 20
- 存在循环依赖（A → B → C → A）
- ORM 在循环中查询数据库（N+1 模式）
- API 接口返回了不应暴露的内部数据字段
- 无故禁用 TLS 证书校验

### 🟡 中危 (Medium)

**判定标准**：满足以下任一条件
- 轻微违反最佳实践或编码约定
- 不影响功能正确性，但可能影响可读性或导致未来 Bug
- 未使用的导入、变量、函数
- 缺少必要的注释或文档字符串
- 命名不符合项目约定

**响应时限**：1 月内
**报告要求**：简要描述即可

**典型示例**：
- 未使用的 import 语句
- 魔术数字直接写在代码中
- 函数缺少类型标注
- 异常被捕获但未做任何处理（空 catch）

### 🔵 低危 (Low)

**判定标准**：满足以下任一条件
- 观察性发现，不构成风险但需注意
- 可能存在改进空间，但当前实现无问题
- 库存在新版本可用
- 文档记录与实际情况有偏差

**响应时限**：酌情处理
**报告要求**：注明"仅供参考"

**典型示例**：
- 依赖包有更新的次要版本可用
- 文件名命名风格与其他文件不一致
- 存在 TODO 注释但无实际影响

### ⚪ 信息 (Info)

**判定标准**：满足以下任一条件
- 观察性发现，不构成风险
- 可能存在改进空间，但当前实现无问题
- 库存在新版本可用
- 文档记录与实际情况有偏差

**响应时限**：酌情处理
**报告要求**：注明"仅供参考"

**典型示例**：
- 依赖包有更新的次要版本可用
- 文件名命名风格与其他文件不一致
- 存在 TODO 注释但无实际影响

## 特殊情况处理

### 无法确认的发现项

当证据不足以确定严重等级时：
1. 标记为临时等级（加 `?` 后缀，如 `🔴?`）
2. 在描述中注明不确定因素
3. 建议人工复核

### 跨维度发现项

涉及多个维度的发现项（如一个安全漏洞因架构设计不良导致）：
1. 按最高的严重等级分类
2. 在维度字段标记为 `{主维度}/{次维度}`（如 `安全/架构`）
3. 在详细分析中说明跨维度关系

### 批量发现模式

当同一模式在多个位置出现时：
1. 不逐条列报，而是汇总为一条发现项
2. 在描述中说明影响范围和出现次数
3. 严重等级取决于模式本身的危害程度

### 误报风险判定

| 确信度 | 标签 | 处理方式 |
|--------|------|---------|
| 高度确信（>90%） | 正常报告 | 无需额外说明 |
| 中等确信（60-90%） | `[需确认]` | 标注需人工复核 |
| 低确信（<60%） | `[疑似]` | 列入信息等级，不应作为行动项 |

## 等级晋升/降级规则

| 情况 | 调整 |
|------|------|
| 发现项同时满足多个等级条件 | 按最高等级 |
| 发现项位于生产环境关键路径 | 升一级 |
| 发现项影响用户隐私或合规要求 | 升一级 |
| 发现项已存在超过一个发布周期未修复 | 升一级 |
| 发现项仅在测试/开发代码中存在 | 降一级 |
| 发现项在官方文档中被列为已知限制 | 降一级 |
