# 六维审计总纲

本文档定义六个审计维度的检查范围、分析方法、常用工具和典型发现模式。执行审计时按选中维度参考对应章节。

---

## 一、代码质量审计

### 检查范围

| 子项 | 检查方法 | 工具/命令 |
|------|---------|----------|
| 死代码（未使用的导入、变量、函数、类、导出） | 跨文件引用分析 | `Grep` 跨文件搜索引用 |
| 代码重复 | 模式识别 | `Grep` 搜索相似代码块 |
| 圈复杂度 | 分支/循环/异常处理计数 | `Glob` + 手动分析 |
| 命名约定 | 检查命名风格一致性 | `Grep` 按命名模式搜索 |
| 注释质量 | 检查缺失/过时/误导性注释 | `Grep` 搜索 TODO/FIXME/HACK |
| 类型安全 | 检查 `any`、类型断言、未类型化参数 | `Grep` 搜索类型盲点 |
| 错误处理完整性 | 检查空 catch、忽略的错误返回 | `Grep` 搜索 catch/error 模式 |
| 资源泄漏 | 检查文件、连接、监听器是否释放 | `Glob` 识别资源分配点 |

### 具体检查项

#### 死代码检测
- [ ] 检查未使用的 import/using/include 语句
- [ ] 检查定义了但未被调用的函数和类
- [ ] 检查从未被读取的变量赋值
- [ ] 检查导出但未被外部引用的模块成员
- [ ] 检查注释掉的代码块（超过 5 行）
- [ ] 检查条件恒为 false/true 的分支（如 `if (false)`）

#### 代码重复检测
- [ ] 搜索连续重复的代码块（相同或极相似的结构）
- [ ] 检查复制粘贴的 API 调用模式
- [ ] 检查多处重复的常量定义和配置值
- [ ] 检查重复的错误处理逻辑

#### 圈复杂度检查
- [ ] 函数内 if/else/else if 分支数 > 5 → 标记
- [ ] 函数内 switch case 分支数 > 10 → 标记
- [ ] 嵌套循环/条件层次 > 3 层 → 标记
- [ ] 函数内 try-catch 块数 > 3 → 标记
- [ ] 包含多个 and/or 逻辑运算符的复合条件 → 标记

#### 命名约定检查
- [ ] 检查是否遵循项目约定的命名风格（camelCase/snake_case/PascalCase）
- [ ] 检查缩写是否一致
- [ ] 检查布尔变量是否以 is/has/should/can 开头
- [ ] 检查函数名是否以动词开头
- [ ] 检查类名是否为名词短语

#### 注释质量检查
- [ ] 检查公开 API 函数是否缺少文档注释
- [ ] 检查注释与代码是否同步（注释描述的行为与实际不符）
- [ ] 检查遗留/FIXME/HACK/TODO 注释数量
- [ ] 检查被注释掉的调试代码

#### 类型安全
- [ ] 检查 `any` 类型的使用（TypeScript）
- [ ] 检查未经校验的类型断言（`as T`）
- [ ] 检查未标注返回值类型的函数
- [ ] 检查未标注参数类型的函数
- [ ] 检查隐式 `any`

#### 错误处理
- [ ] 检查空的 catch 块
- [ ] 检查 catch 后仅打印日志未做处理
- [ ] 检查忽略的错误返回值
- [ ] 检查未处理的 Promise rejection
- [ ] 检查吞没异常的 try-finally 结构
- [ ] 检查未正确处理边界条件（空列表、null、undefined）

#### 资源泄漏
- [ ] 检查文件流是否在 finally/using 中关闭
- [ ] 检查数据库连接是否在 finally 中释放
- [ ] 检查事件监听器是否在组件销毁时移除
- [ ] 检查定时器/间隔是否在适当时机清除
- [ ] 检查锁是否在异常时释放

## 二、安全审计

### 检查范围

按 OWASP Top 10 (2021) 逐项排查：

| 类别 | 漏洞类型 | 检查要点 |
|------|---------|---------|
| A01 | 访问控制失效 | 越权、路径遍历、权限提升 |
| A02 | 加密机制失效 | 敏感数据明文传输/存储、弱加密算法 |
| A03 | 注入 | SQL/NoSQL/命令/代码/模板注入 |
| A04 | 不安全设计 | 缺少安全控制、速率限制 |
| A05 | 安全配置错误 | 默认凭证、错误信息泄露、未禁用调试 |
| A06 | 易受攻击和过时的组件 | 已知漏洞依赖 |
| A07 | 识别和认证失效 | 弱密码、会话固定、Token 存储 |
| A08 | 软件和数据完整性失效 | 不安全的反序列化、CI/CD 管道 |
| A09 | 安全日志记录和监控失效 | 缺少审计日志、日志泄露敏感信息 |
| A10 | 服务端请求伪造 (SSRF) | URL 未校验、内网访问 |

### 具体检查项

#### A01 - 访问控制
- [ ] 检查 API 路由是否有鉴权中间件
- [ ] 检查是否在服务端做权限校验（而非仅前端隐藏）
- [ ] 检查 IDOR 模式（用户可指定他人资源的 ID）
- [ ] 检查路径拼接是否做了规范化（防止 `../` 遍历）
- [ ] 检查管理接口是否暴露在公网

#### A02 - 加密机制
- [ ] 检查密码是否使用 bcrypt/argon2/scrypt（非 MD5/SHA1）
- [ ] 检查是否使用了 HTTPS（而非 HTTP）
- [ ] 检查是否硬编码了密钥/密码/Token
- [ ] 检查是否使用了已废弃的加密算法（DES/RC4/MD5/SHA1）
- [ ] 检查传输敏感数据时是否加密

#### A03 - 注入
- [ ] 检查 SQL 查询是否使用参数化查询（非字符串拼接）
- [ ] 检查 NoSQL 查询是否过滤特殊操作符（`$where`、`$gt`）
- [ ] 检查 `eval()`、`setTimeout(string)` 等动态执行
- [ ] 检查 shell 命令是否使用了未经处理的用户输入
- [ ] 检查模板引擎是否使用了用户输入的 raw/unsafe 输出

#### A04 - 不安全设计
- [ ] 检查是否有速率限制（Rate Limiting）
- [ ] 检查是否有请求验证（非仅前端校验）
- [ ] 检查是否有账户锁定机制
- [ ] 检查是否有 CSRF 防护

#### A05 - 安全配置
- [ ] 检查默认密码是否已修改
- [ ] 检查 CORS 配置是否过于宽松（`Access-Control-Allow-Origin: *`）
- [ ] 检查调试模式是否已禁用
- [ ] 检查错误页面是否暴露调用栈和路径信息
- [ ] 检查 HTTP 安全头（CSP、HSTS、X-Frame-Options）

#### A06 - 过时组件
- [ ] 检查 package.json/yarn.lock 中是否有过时版本
- [ ] 检查是否有已知 CVE 的依赖
- [ ] 检查废弃/不再维护的库
- [ ] 检查许可证兼容性

#### A07 - 认证与会话
- [ ] 检查密码策略（最小长度、复杂度要求）
- [ ] 检查 JWT/会话 Token 的存储方式（避免 localStorage）
- [ ] 检查会话过期时间
- [ ] 检查 Token 是否在注销时失效
- [ ] 检查是否有暴力破解防护

#### A08 - 数据完整性
- [ ] 检查反序列化是否使用了安全的类型约束
- [ ] 检查文件上传的类型校验（MIME、Magic bytes）
- [ ] 检查 CI/CD 脚本是否验证了工件的签名

#### A09 - 日志与监控
- [ ] 检查日志是否包含密码、密钥、敏感信息
- [ ] 检查是否有安全事件的审计日志
- [ ] 检查日志是否集中管理

#### A10 - SSRF
- [ ] 检查 URL 输入是否经过白名单校验
- [ ] 检查内部服务发现是否限制网络访问
- [ ] 检查是否防止了内网 IP 访问（127.0.0.1、10.x.x.x 等）

## 三、架构审计

### 检查范围

| 子项 | 检查方法 | 说明 |
|------|---------|------|
| 分层违规 | 分析导入/引用路径 | 展示层是否直接操作数据库？ |
| 循环依赖 | 分析模块引用图 | A→B→C→A |
| 接口滥用 | 检查接口定义和使用 | 是否使用了接口？接口粒度是否合适？ |
| 单一职责 | 分析类/模块的大小和职责 | 类是否做了太多不同的事？ |
| 抽象程度 | 评估抽象层次 | 过度抽象 vs 抽象不足 |
| API 设计 | 检查 API 路由和响应 | RESTful 命名、状态码、错误格式一致性 |

### 具体检查项

#### 分层违规
- [ ] 检查 Controller/Handler/Route 层是否直接操作数据库
- [ ] 检查展示层是否引入了数据访问层的依赖
- [ ] 检查业务逻辑是否混入展示层
- [ ] 检查工具函数是否跨越了预期边界

#### 循环依赖
- [ ] 检查模块 A 导入 B，B 导入 A 的模式
- [ ] 检查跨层级的间接循环引用
- [ ] 检查接口与实现之间的循环依赖

#### 接口设计
- [ ] 检查是否对抽象（接口/基类）编程而非具体实现
- [ ] 检查接口的粒度是否合适（接口隔离原则）
- [ ] 检查是否违反了里氏替换原则
- [ ] 检查接口是否包含了调用方不需要的方法

#### 单一职责
- [ ] 检查类/文件是否超过 400 行
- [ ] 检查函数是否超过 50 行
- [ ] 检查类是否操作了不相关的数据
- [ ] 检查类是否包含多个独立职责的 public 方法组

#### 抽象程度
- [ ] 检查是否存在不必要的工厂/抽象层（过度工程）
- [ ] 检查是否存在大量相似代码未被抽象（抽象不足）
- [ ] 检查抽象层是否正确隐藏了实现细节
- [ ] 检查是否有"上帝类"或"万能类"

#### API 一致性
- [ ] 检查 RESTful 路由命名风格是否一致
- [ ] 检查 HTTP 状态码使用是否正确
- [ ] 检查错误响应格式是否统一
- [ ] 检查 API 版本化策略

## 四、性能审计

### 检查范围

| 子项 | 检查方法 | 说明 |
|------|---------|------|
| N+1 查询 | 扫描循环内的数据库/API 调用 | 循环中是否存在查询？ |
| 内存泄漏 | 检查资源生命周期 | 闭包引用、全局缓存、事件监听器 |
| 不必要分配 | 检查热路径中的对象创建 | 循环内 `new`、临时对象 |
| 低效算法 | 检查复杂度和冗余操作 | 嵌套循环、未优化集合操作 |
| 缓存机会 | 识别可缓存的计算 | 重复计算、不变数据 |

### 具体检查项

#### N+1 查询
- [ ] 检查 for/forEach/map 循环中是否包含 await 数据库查询
- [ ] 检查 ORM 的 lazy loading 是否导致了隐含的循环查询
- [ ] 检查批量操作是否逐条处理而非批量处理
- [ ] 检查 API 调用是否在循环中发起

#### 内存泄漏
- [ ] 检查全局变量/缓存是否无限增长（无淘汰策略）
- [ ] 检查事件监听器是否在组件卸载时移除
- [ ] 检查定时器/间隔是否在不需要时清除
- [ ] 检查闭包是否持有了大对象引用
- [ ] 检查 `addEventListener` 是否有对应的 `removeEventListener`

#### 不必要分配
- [ ] 检查循环体内是否创建了新的对象/数组
- [ ] 检查循环内是否有字符串拼接（应使用 StringBuilder/Array.join）
- [ ] 检查频繁调用的函数内是否有临时对象创建
- [ ] 检查是否在热路径中进行了正则表达式编译

#### 低效算法
- [ ] 检查嵌套循环（O(n²) 及以上复杂度）
- [ ] 检查未使用 HashMap/Set 的线性搜索
- [ ] 检查大数组的频繁插入/删除操作
- [ ] 检查未使用索引的数据库查询
- [ ] 检查不必要的深拷贝

#### 缓存机会
- [ ] 检查相同输入的重复计算
- [ ] 检查不变数据的重复读取
- [ ] 检查可预热的查询或计算结果
- [ ] 检查是否有合适的缓存策略（TTL、LRU、写穿透）

## 五、依赖审计

### 检查范围

| 子项 | 检查方法 | 说明 |
|------|---------|------|
| 过时版本 | 检查依赖版本与最新版 | 是否使用了过时的大版本？ |
| 已知漏洞 | 查找已知 CVE | `npm audit`、`cargo audit` 等 |
| 废弃包 | 检查包是否被标记为废弃 | npm deprecate、pypi 状态 |
| 未使用依赖 | 检查导入/使用情况 | `depcheck` 或手动检查 |
| 依赖规模 | 评估依赖引入的代码量 | 是否引入了大型库仅使用小功能？ |

### 具体检查项

- [ ] 检查 package.json/requirements.txt/Cargo.toml 等依赖声明文件
- [ ] 检查 lock 文件（package-lock.json、yarn.lock、Cargo.lock）是否在版本控制中
- [ ] 检查依赖版本是否固定（而非 `^` 范围）
- [ ] 检查是否有已知 CVE 的依赖
- [ ] 检查是否有不再维护的依赖
- [ ] 检查是否有未直接使用但存在于依赖树中的包
- [ ] 检查 devDependencies 与 dependencies 是否混淆
- [ ] 检查是否引入了大型库仅使用其一小部分功能（建议用轻量替代）

## 六、合规审计

### 检查范围

| 子项 | 检查方法 | 说明 |
|------|---------|------|
| 许可合规 | 检查 LICENSE 文件和依赖许可 | 开源许可证兼容性 |
| 编码标准 | 检查是否遵循项目编码规范 | 与 CLAUDE.md 的编码标准对比 |
| 日志标准 | 检查日志格式和内容 | 是否统一日志框架？是否脱敏？ |
| 数据隐私 | 检查个人数据处理 | GDPR/个人信息保护法 |
| API 版本化 | 检查 API 兼容性声明 | 是否遵循语义化版本？ |

### 具体检查项

#### 许可合规
- [ ] 检查根目录是否有 LICENSE 文件
- [ ] 检查依赖的许可证是否与项目兼容
- [ ] 检查是否包含 GPL/AGPL 等可能传染的许可证

#### 编码标准
- [ ] 检查是否与项目 CLAUDE.md 中定义的编码标准一致
- [ ] 检查 linter 配置是否存在且被遵循
- [ ] 检查格式化器配置是否存在（prettier、rustfmt 等）

#### 日志标准
- [ ] 检查日志是否包含敏感信息（密码、Token、身份证号）
- [ ] 检查日志级别使用是否合理（info/warn/error）
- [ ] 检查是否使用了统一的日志框架
- [ ] 检查异常日志是否包含足够的上下文信息

#### 数据隐私
- [ ] 检查用户个人数据的收集和存储是否有声明
- [ ] 检查是否有数据删除/匿名化机制
- [ ] 检查 Cookie/存储的使用是否需要用户同意

#### API 版本化
- [ ] 检查公开 API 是否有版本声明
- [ ] 检查是否遵循语义化版本 (SemVer)
- [ ] 检查是否提供了向后兼容性
